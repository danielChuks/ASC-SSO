# Paper Analysis: Anonymous Self-Credentials and their Application to Single-Sign-On

*Alupotha, Barbaraci, Kaklamanis, Rawat, Cachin, Zhang — ePrint 2025/618, April 2025*

---

## 1. Core Problem

The paper addresses a fundamental tension in digital identity systems:

- **Sybil resistance** requires that a single entity cannot appear as multiple entities. Service providers need this to fairly allocate resources and prevent abuse.
- **Unlinkability** (privacy) requires that a user's activity across different services cannot be correlated. Users need this to prevent tracking.

These appear contradictory. Sybil resistance pushes toward single, verifiable identities. Unlinkability pushes toward multiple, disconnected pseudonyms. The paper's contribution is a construction that achieves both simultaneously, without trusting a central identity provider.

---

## 2. Motivation: Why Existing Approaches Fall Short

The paper identifies two broad categories of prior work, each with a fatal flaw:

**IdP-based systems** (OAuth, OIDC, Idemix, CanDID): A trusted Identity Provider issues credentials to users. Privacy depends on the IdP not tracking which users access which services. The paper argues this trust assumption is no longer appropriate: IdP data has become a commercial asset, IdPs can be coerced by authorities, and even distributed IdP schemes (threshold-based) leak timing information through the pattern of when users interact with both the IdP and the service.

**Self-sovereign identity on blockchains** (DID, uPort, EBSI): An immutable public registry stores identities. Users publish pseudonyms on the registry and share them with services. The problem: blockchain transactions are traceable. If a user pays to register pseudonyms, those payments are linkable. The paper notes that untraceable blockchains (Monero-style) exist but are too slow for large-scale identity management.

The paper's proposed solution: publish only a **master identity** (a cryptographic commitment) on the registry. Generate per-service pseudonyms locally. Prove — using a zero-knowledge argument — that a pseudonym was generated by someone whose master identity appears in the public registry, without revealing which one.

---

## 3. The ASC Notion: A Formal Framework

The paper introduces Anonymous Self-Credentials (ASC) as a formal cryptographic primitive. An ASC system has:

- **N provers**, each holding a master credential `(Φ_j, sk_j)` — a public master identity and a secret key
- **L verifiers**, each with a unique verifier identifier `v_l`
- An **anonymity set** `Λ = [Φ_1, ..., Φ_N]` — the set of all master identities, public to all

The ASC protocol defines four algorithms:

- `ASC.Setup(λ, L)` → generates public parameters including all verifier identifiers
- `ASC.Gen(crs, sk)` → prover self-generates a master identity from a secret key
- `ASC.Prove(crs, Λ, l, j, sk_j, ϕ)` → produces `(nullifier, proof)` for pseudonym `ϕ` at verifier `v_l`
- `ASC.Verify(crs, Λ, l, ϕ, nul, π)` → verifier checks that `(ϕ, nul, π)` is valid

**Key insight on nullifiers**: The nullifier `nul` is deterministically derived from `(sk_j, v_l)`. This means:
- Same prover, same verifier → always same nullifier → Sybil resistance (second registration is rejected)
- Same prover, different verifiers → different nullifiers → registrations are unlinkable across verifiers
- Different provers, same verifier → different nullifiers → multiple honest registrations coexist

---

## 4. Security Properties (Formally Defined)

The paper provides game-based definitions for all five security properties:

**Correctness**: Honest provers can always register with any verifier; their nullifiers match the deterministic derivation.

**Robustness**: Even if the anonymity set contains maliciously crafted fake identities, an honest prover can still generate valid proofs. This models the real-world scenario where a permissionless registry may accumulate garbage entries.

**Unforgeability** (Strong): An adversary who controls all master identities except one honest prover's cannot forge a proof for a fresh pseudonym on behalf of that prover, even after observing many genuine proofs. "Fresh pseudonym" is key — the adversary cannot reuse proof material from previous interactions.

**Sybil Resistance**: A master identity can produce at most one distinct nullifier for any given verifier. An adversary who controls all master identities cannot produce two different valid nullifiers bound to the same master identity and same verifier.

**Anonymity**: Given a nullifier and proof, a computationally bounded adversary cannot determine which master identity in the anonymity set generated them, as long as at least two honest provers exist. The anonymity set provides a k-anonymity guarantee with `k = N - N'` where `N'` is the number of corrupted provers.

**Multi-Verifier Unlinkability**: Given two `(proof, nullifier)` pairs for two different verifiers, an adversary cannot determine whether they came from the same prover or two different provers, even if both verifiers collude. This is the formal guarantee against cross-service tracking.

## 5. Two Concrete Constructions

The paper presents two implementations of ASC, using different zero-knowledge argument systems:

### SRS-ASC (Semaphore / zk-SNARK based)

The master identity is a commitment `C_j = Com(sk_j)` using an algebraic commitment scheme that requires a Structured Reference String (SRS).

The nullifier is computed as `nul = Hash_srs(sk_j, v_l)` using a ZK-friendly algebraic hash (Poseidon3 in the implementation). The ZK circuit proves:
1. The prover knows `(j, sk_j)` such that `C_j ∈ Λ` (membership)
2. The nullifier was correctly computed from `sk_j` and `v_l`

**Performance** (from paper, N=1024):
- Proof size: 328 bytes (constant, independent of N)
- Proof generation: ~200ms (logarithmic in N)
- Proof verification: ~10ms (logarithmic in N)
- Master identity size: 96 bytes

**Trusted setup required**: The SRS must be generated by a trusted ceremony. This is the key limitation.

### CRS-ASC (Bulletproof / range-proof based)

The master identity is a multi-value commitment `C_j = Com([nul_1, nul_2, ..., nul_L], k_j)` that commits to all L nullifiers simultaneously along with a blinding key.

The nullifiers are pre-generated during identity creation (using HKDF) and committed into the master identity. During registration, the prover reveals the nullifier for verifier `l` and proves it is correctly committed in `C_j`.

This construction uses Pedersen commitments on SECP256K1 and a Bootle et al. membership proof — both based on the standard Discrete Log assumption with no trusted setup (only a Common Reference String).

**Performance** (from paper, N=1024):
- Proof size: ~4KB (logarithmic in N)
- Proof generation: ~500ms (linear in N)
- Proof verification: ~80ms (linear in N)
- Master identity size: 33 bytes

**No trusted setup required**: This is the key advantage.

## 6. The U2SSO System

U2SSO is the full credential management system built on top of ASC. It adds:

**HKDF-based child credential derivation**: The user has a master secret `(sk, r)`. For each service `v_l`, they derive a child secret `csk_l = HKDF(r, v_l)`. The pseudonym `ϕ = Gauth.Gen(csk_l)` is the public key of this child credential. This derivation is deterministic — the user can always recompute `csk_l` from `r` without storing it separately.

**Two-phase operation**:
- **Registration** (infrequent, uses ASC proof): User sends `(ϕ, nul, π)` to the SP. SP verifies ZK proof and checks nullifier. This is the expensive, privacy-sensitive operation.
- **Authentication** (frequent, uses only Gauth): User responds to a challenge with `σ = Gauth.Prove(csk_l, W)`. No ZK proof, no IdR interaction. This is fast and has the same security as any challenge-response signature scheme.

**Identity Registry (IdR)**: An Ethereum smart contract stores master identities. Users pay a gas fee to register, which provides Sybil resistance (registering multiple identities is costly). The contract stores identities in fixed-size anonymity sets `Λ_1, Λ_2, ..., Λ_d` — the user learns which set their identity is in when the set is finalized.

**Timing attack resistance**: Because authentication does not require IdR interaction, and registration uses fixed-size anonymity sets (not real-time queries), there is no per-login timing signal that colluding SPs and the IdR could use to de-anonymize users.

## 8. Trust Assumptions

- **IdR (smart contract)**: May collude with SPs. Cannot deanonymize users due to the ZK proofs. Must be available and consistent (blockchain guarantees this).
- **Anonymity set size**: Privacy holds as long as at least 2 honest provers are in the set. The paper uses N >> 2 (e.g., N=1024) to provide meaningful anonymity.
- **SRS-ASC trusted setup**: The SRS ceremony must be trusted. If the internal randomness of the SRS generation is leaked, fraudulent proofs can be created. The paper uses Semaphore's existing SRS — a well-audited trusted ceremony.
- **CRS-ASC**: No trusted setup. Security rests on the Discrete Log assumption over SECP256K1.

---

## 9. Strengths for the DAO Voting Use Case

- **Self-sovereignty**: Voters generate their own credentials; no governance committee or protocol team can censor registrations
- **Unlinkability across proposals**: The multi-verifier unlinkability property directly maps to cross-proposal vote unlinkability
- **Performance**: Semaphore-based proofs generate in ~200ms on consumer hardware — viable for browser-side proof generation
- **Proven formal security**: The construction has published proofs, not just intuitive arguments
- **Nullifier-native design**: Sybil resistance (one-vote-per-member) is a first-class protocol property, not an afterthought

---

## 10. Limitations and Weaknesses

- **Static anonymity sets**: The anonymity set is fixed when finalized. A late registrant must wait for the next set. This creates a delay between token acquisition and voting eligibility.
- **No token-balance weighting**: The basic ASC construction gives every registered identity one vote. Token-weighted voting (where 1000 tokens = 1000 votes) is not directly modeled.
- **No on-chain ZK verification** in the PoC: The reference implementation uses a Node.js subprocess to verify Semaphore proofs off-chain. A production system needs a Solidity Groth16 verifier.
- **Trusted setup (SRS-ASC)**: The Semaphore SRS ceremony is required. CRS-ASC avoids this but has larger, slower proofs.
- **No revocation of master identities**: The paper describes pseudonym revocation, but revoking a master identity from an anonymity set is complex and not implemented.
- **Ethereum-specific Sybil resistance**: The financial cost of identity registration is an imperfect Sybil control — it depends on gas prices, token prices, and rational-actor assumptions.
