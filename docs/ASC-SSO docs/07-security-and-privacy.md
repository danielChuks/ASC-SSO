## 1. Privacy Guarantees

### 1.1 Voter Anonymity 

**Guarantee**: Given a valid vote submission `(proof, nullifier, vote_choice, merkle_root)`, a computationally bounded adversary cannot determine which registered commitment produced the proof.

**Source**: The Semaphore Groth16 ZK proof has the Special Honest-Verifier Zero-Knowledge (SHVZK) property (Definition 5 from the paper). The proof reveals nothing about the witness `(secret, merkle_path)` beyond what is already public (the valid Merkle root and a correctly derived nullifier).

**Minimum condition**: The anonymity set must contain at least 2 honest voters. If only one voter has registered, the proof trivially identifies them. In practice, anonymity scales with set size N — the paper guarantees the adversary's advantage is bounded by `1/N + ε(λ)` where N is the number of honest provers.

**What "honest" means**: A voter's commitment was generated correctly using `commitment = Poseidon(BabyJubjub_pubkey(secret))`. A commitment generated by an adversary who chose a `secret` they control counts as a corrupted prover.

### 1.2 Cross-Proposal Unlinkability

**Guarantee**: Given two vote submissions from the same voter for proposals A and B, an adversary — including a colluding backend and registry — cannot determine they came from the same voter.

**Source**: 
- `nullifier_A = Poseidon([A, secret])` and `nullifier_B = Poseidon([B, secret])` are computationally independent, as Poseidon is a ZK-friendly cryptographic hash.
- `vote_pubkey_A = Ed25519_pubkey(HKDF(r, A))` and `vote_pubkey_B = Ed25519_pubkey(HKDF(r, B))` are unlinkable by the HKDF unlinkability definition (Definition 1 from the paper).

This is the multi-verifier unlinkability property: proposals play the role of verifiers.

### 1.3 On-Chain Privacy

**Guarantee**: The blockchain (Ethereum) records only the voter's commitment (a Poseidon hash). It does not record any vote choices, nullifiers, or ZK proofs.

**What is public on-chain in the PoC**: Only the commitment in `CommitmentRegistry`. Vote submissions go to the backend (off-chain). In the production path (DAOVoting.sol), nullifiers and vote choices would be on-chain but not linked to voter identity.

## 2. Integrity Guarantees

### 2.1 Vote Authenticity (Unforgeability)

**Guarantee**: An adversary who does not know a voter's `secret` cannot generate a valid ZK proof claiming to be in the anonymity set.

**Source**: The Groth16 knowledge soundness property (computational witness-extended emulation, Definition 4 from the paper). An extractor exists that, given any valid proof, can extract a witness `(secret, merkle_path)`. Therefore, a valid proof implies knowledge of the secret.

**In practice**: Forgery requires breaking the Groth16 proof system, which rests on the KoE assumption over the BN254 curve. This is considered secure at the 128-bit security level.

### 2.2 Tally Integrity

**Guarantee**: The tally reflects all submitted votes exactly once. No vote is omitted, duplicated, or modified after submission.

**In the PoC**: Tally integrity is enforced by the PostgreSQL database. The backend is the trusted tallier. This is a limitation of the PoC.

**In production**: Tally integrity is enforced by the smart contract. Vote choices and nullifiers are stored on-chain; the tally is computed by reading immutable on-chain state. This removes the need to trust the backend.

### 2.3 Vote Choice Binding (Partial in PoC)

**In current PoC**: The vote choice is sent as a separate field alongside the ZK proof. The proof does not cryptographically bind the vote choice — an adversary who intercepts the POST request could modify `vote_choice` without invalidating the proof.

**Mitigation in PoC**: HTTPS transport prevents casual MITM. The pseudonym (Ed25519 public key) is derived from `(r, proposal_id)` — a second vote attempt with a different choice would use the same pseudonym and be rejected by the pseudonym uniqueness check.

**Production fix**: Encode `vote_choice` in the Semaphore `message` field. The message is committed inside the Groth16 proof — modification is computationally infeasible.

## 3. Double-Voting Prevention

The nullifier mechanism provides the primary defense:

```
nullifier = Poseidon([proposal_id, secret])
```

**Properties**:
- Same voter, same proposal → identical nullifier → second vote rejected
- The nullifier is computed deterministically — a voter cannot compute two different nullifiers for the same proposal
- This is the Sybil resistance property from the paper (Definition 10)

**Second line of defense**: The pseudonym uniqueness check in the backend. A voter derives a deterministic `vote_pubkey = Ed25519_pubkey(HKDF(r, proposal_id))` for each proposal. Attempting to vote twice produces the same `vote_pubkey`, which is also rejected.

**Limitation**: In the current PoC, the nullifier registry is in PostgreSQL (the backend is trusted). In production, the nullifier registry is in the smart contract — rejection is enforced by consensus.

## 4. Replay Resistance

**Scenario**: An attacker intercepts a valid vote submission and replays it against a different proposal or re-submits it.

**Same proposal replay**: The nullifier check rejects it — `(proposal_id, nullifier)` already exists.

**Different proposal replay**: The nullifier will not match. `nullifier = Poseidon([proposal_id, secret])` — the same proof with a different `proposal_id` in the scope input would be a different circuit computation and would fail the Groth16 verification (the `scope` is a public input committed in the proof).

**Gauth replay**: The challenge-response mechanism uses one-time challenges with 5-minute TTL and a `used` flag. A replayed authentication signature will fail the `used != 0` check.

## 5. Sybil-Related Limitations

**Sybil resistance in registration**: A voter can register multiple Semaphore identities by paying multiple gas fees. Each registered commitment counts as a separate member of the anonymity set and gets one vote. Economic cost (gas) is the only deterrent — there is no identity verification.

**Impact on governance**: In a token-weighted DAO, vote weight should reflect token balance. The basic ASC-SSO system gives one vote per commitment, not one vote per token. A wealthy attacker could register many commitments and use each to vote, concentrating voting power.

**Mitigations** (not implemented in PoC):
- Token-snapshot-gated registration: require proof of token ownership at a snapshot block
- On-chain Sybil resistance: require each commitment to be bonded with a token amount; the bond is slashed if double-voting is detected
- Social layer: DAO constitution specifies rules for eligibility; on-chain enforcement is a separate module

---

## 6. Token Snapshot Assumptions

The PoC does not implement token-snapshot-based eligibility. The anonymity set is simply all commitments in `CommitmentRegistry` — any Ethereum wallet can register.

A production DAO deployment would need:
1. An eligibility snapshot at block `B_snapshot`
2. A proof that the voter's commitment was registered by a wallet holding `≥ threshold` tokens at block `B_snapshot`
3. This proof would be an additional ZK circuit: a Semaphore-style membership proof combined with a Merkle proof of the ERC-20 state at the snapshot block

## 7. ZK Proof System Trust Assumptions

**Groth16 (SRS-ASC)**:
- Requires a trusted setup ceremony to generate the SRS (structured reference string = the `.zkey` file)
- If the internal randomness of the SRS generation is leaked, an adversary can generate proofs for false statements
- Mitigation: Use the existing Semaphore trusted setup, which involved multiple independent parties. The U2SSO repo ships pre-generated `.zkey` files for N ∈ {16,...,1024}

**BN254 curve**:
- The security of the Groth16 proof system rests on hardness assumptions over the BN254 pairing-friendly curve
- The Knowledge of Exponent (KoE) assumption and the Diffie-Hellman assumption over BN254 are standard and widely accepted at 128-bit security

**Poseidon hash**:
- Used for the identity commitment and the nullifier inside the circuit
- Designed specifically for ZK-friendliness; resistance against known attacks is well-studied

## 8. Threat Model Summary

| Threat | Attack | Defense |
|---|---|---|
| Vote forging | Adversary creates proof without being in anonymity set | Groth16 soundness (KoE assumption) |
| Double voting | Voter submits two votes for same proposal | Nullifier uniqueness check (deterministic) |
| Voter identity exposure | Adversary links nullifier to voter commitment | ZK anonymity (SHVZK property) |
| Cross-proposal linkage | Link votes across proposals to build voter profile | Multi-verifier unlinkability (HKDF + ZK) |
| Vote manipulation in transit | MITM changes `vote_choice` | HTTPS (PoC); ZK message binding (production) |
| Anonymity set gaming | Adversary registers many fake commitments | Economic cost (gas); does not harm anonymity of honest voters (robustness property) |
| Replay attack | Reuse a valid proof for a different proposal | Scope is committed in proof; wrong scope → invalid proof |
| Timing correlation | Correlate registry read times with vote submission times | Static anonymity set; no per-vote IdR interaction |
| Backend compromise | Backend lies about tally or modifies votes | Move to on-chain tally (production path) |
| Sybil voter registration | Register many commitments to amplify vote weight | Economic cost + token-snapshot eligibility (production) |

## 9. Out-of-Scope Risks (PoC Specific)

- **Backend trust**: In the PoC, the backend is trusted. It could silently drop votes, lie about the tally, or expose `(nullifier, vote_choice)` mappings. This is acceptable for a hackathon demo; production requires on-chain enforcement.
- **localStorage security**: The voter's `secret` and `r` are stored in browser `localStorage`. A malicious browser extension or XSS attack could extract them. Production requires hardware key storage or a browser extension wallet.
- **Anonymity set size**: The demo starts with a small number of voters (e.g., 5). Meaningful anonymity requires large sets (N ≥ 64 recommended). The proof system supports up to N=1024 with the pre-built keys.
